### <a name="_qmphm5d6rvi3"></a>**Решение**
Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения. 

[Context-DepositProcess.puml](Context-DepositProcess.puml)
[Container-DepositProcess.puml](Container-DepositProcess.puml)

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.

1. Предлагается ввести в систему новый микросервис API Deposit Service, который бы обеспечил требуемый новый функционал 
по работе со ставками депозитов и с заявками депозитов. В его БД будут храниться депозитные предложения, как общие так и персональные, и заявки на открытие депозитов.
Интернет-банк, Сайт и Система кол-центра обращаются по REST синхронно для того чтобы получить предложения или добавить заявку.
2. Далее вводится новое клиентское приложение Управление Предложениями Депозитов, в который переезжает функцинал xls-файла,
теперь менеджеры бек-офиса работают непосредственно с ним, старая бизнес-логика xls-файла реализуется в API Deposit Service,
таким образом API Deposit Service содержит актуальную информацию о ставках.
3. Сейчас непосредственно АБС отправляет смс, если это создает нагрузку на АБС, то можно ввести отдельный сервис SMS Consumer,
который бы брал их из очереди Kafka и отправлял непосредственно в СМС-шлюз. А в АБС доработать так, чтобы смс только помещать 
в очередь и не заниматься непосредственно отправкой в шлюз. На первом этапе этот новый путь в АБС используется только для нового процесса
- извещения о состоянии заявки на депозит. Старый способ отправки смс в АБС остается и от него отказываются постепенно.
АБС получает заявки на депозиты через очередь Kafka - можно регулировать производительность этой доставки (требования +R4, +R9).
То что АБС в рамках нового процесса взаимодействует с остальными участниками через очереди - немного снимет синхронную нагрузку на
саму АБС и позволит использовать ее мощности для работы с депозитами (требование +R8).
Скорее всего не получится реализовать работу с Kafka из бизнес логики АБС, для этого следует использовать Debezium и Kafka Connect Sink.
4. Работа с Kafka происходит из API Deposit Service и непосредственно Интернет Банк с Kafka не работает (требования +R6, +R7). 
5. Следуя +R1, +R2, +R3 для API Deposit Service выбираем Java/SpringBoot + PostgreSQL.





### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

1. Из API Deposit Service можно работать непосредственно с АБС в синхронном режиме - отправлять заявки на депозиты и отправлять смс.
Но этот способ увеличит синхронную нагрузку на АБС и лишит систему гибкости по регулированию этой нагрузки.
В плане отправки смс этот способ добавит нагрузку, а не снимет ее, таким образом этот вариант не удовлетворяет требованиям.

2. Можно не выделять отдельный API Deposit Service, а встроить требуемый функционал в существующую систему,
но это потребует заказной разработки вне банка.
